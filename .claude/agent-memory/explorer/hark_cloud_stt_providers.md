Verified 2026-07-15 via WebSearch/WebFetch (see citation URLs per section). Hark is pivoting from on-device sherpa-onnx + Parakeet TDT (rejected, see [[hark-stt-stack-risk]]) to BYOK cloud STT with multi-provider adapters. This replaces on-device latency/footprint tradeoffs with per-request network latency and per-provider vocabulary-biasing mechanisms as the new design constraints.

## OpenAI-compatible `/v1/audio/transcriptions` shape

**OpenAI and Groq share the exact same request/response shape** (multipart form: `file`, `model`, `prompt`, `response_format`, `language`; Bearer auth; `prompt` param for vocabulary biasing truncated to 224 tokens for Whisper-family models). One adapter using this shape covers both providers by just swapping base URL + model name + API key. This is likely the single highest-leverage adapter to build first, since it covers a cheap/fast option (Groq) and a quality option (OpenAI) with zero code duplication.

Deepgram, ElevenLabs, and Mistral each have their own distinct shapes (different auth header, different param names for biasing, different upload semantics) and need dedicated adapters.

## Recommendation for lowest release-to-inject latency on 2-15s clips

- **Groq (`whisper-large-v3-turbo`)** is the strongest default for raw speed: 216x realtime inference, cheapest of all providers by far ($0.000667/min), OpenAI-compatible shape. Caveat: 10-second minimum billed length per request (irrelevant to latency, relevant to cost accounting) and no confirmed sub-second p50 latency figure — the 216x claim is throughput, not the network-round-trip latency that matters for push-to-talk. Not yet verified against real short-clip request latency (need to test empirically or find first-byte-of-response benchmarks, not throughput multiplier claims).
- **Deepgram nova-3** is the strongest option if dictionary/keyword biasing quality matters as much as speed: purpose-built `keyterm` biasing mechanism (up to 500 tokens, no retraining) maps naturally onto Hark's user dictionary feature, and Deepgram explicitly markets sub-300ms latency-tuned models (Flux) for conversational turn-taking, which is a closer real-world proxy for "hold key, speak, release" latency than any other provider's marketing claims.
- Recommended adapter set: **openai-compatible (covers OpenAI + Groq) + deepgram** as the two Phase-1 adapters. This gives one price/quality range (Groq cheap+fast, OpenAI slower+higher quality) plus the best-in-class biasing story (Deepgram) with only two adapter implementations. ElevenLabs/Mistral/AssemblyAI/Speechmatics/Gladia are viable later additions but none clearly beats this pair on the "why" (dictionary biasing + short-clip latency) — add them opportunistically, not as Phase-1 scope.

## Dictionary/vocabulary biasing mapping (Hark's user dictionary → provider mechanism)

- **OpenAI / Groq (Whisper-family):** free-text `prompt` param, 224-token cap on whisper-1/Groq turbo (larger for GPT-4o models). Hark's dictionary would need to be serialized into a short natural-language hint string ("the user may say: <term1>, <term2>, ..."), not a structured list — imprecise compared to Deepgram's mechanism, and 224 tokens is a hard ceiling if the dictionary grows large.
- **Deepgram nova-3/Flux:** `keyterm` param (Nova-3+) is the closest structural match to a dictionary — repeatable param per term, up to 500 tokens total, no weighting. Legacy `keywords` (Nova-2 and earlier, weighted `TERM:WEIGHT` syntax) is NOT available on nova-3, so don't assume weight-based biasing works if Hark ever pins to nova-3.
- **ElevenLabs:** "keyterm prompting" exists as a paid add-on ($0.05/hr) but exact param shape/limits not confirmed this pass — verify before building the adapter.
- **Mistral Voxtral:** batch model claims "up to 100 words/phrases" for spelling guidance, but exact API param name/shape unconfirmed this pass — verify during spike, don't assume parity with OpenAI's `prompt`.

## Gotchas

- **OpenAI/Groq max file size is 25 MB** — irrelevant for 2-15s 16kHz mono WAV clips (well under a few hundred KB), not a practical constraint for Hark's push-to-talk use case.
- **Groq bills a 10-second minimum per request** regardless of actual clip length — for many short push-to-talk utterances (2-5s), Hark will effectively pay for 10s every time. Worth flagging in cost estimates; batching isn't compatible with push-to-talk's one-shot latency requirement.
- **Deepgram's Rust crate provenance is ambiguous** ("community" on GitHub/crates.io vs "official" on docs.rs) — verify current maintenance status and pin an exact version before depending on it; consider hand-rolling the REST call instead given it's a simple multipart/HTTP call, avoiding a third-party crate dependency question entirely.
- **Deepgram's `keyterm` vs `keywords` are mutually exclusive by model generation** — nova-3 only understands `keyterm` (no weighting); don't port nova-2-style weighted keyword lists forward without translation.
- **Realtime/streaming APIs (OpenAI Realtime, Deepgram Flux, ElevenLabs Scribe v2 Realtime, Mistral Voxtral Realtime) are likely unnecessary for Hark's architecture** — push-to-talk captures a bounded utterance (2-15s) before any request is made, so a single batch/prerecorded REST call per utterance is the natural fit, not a persistent streaming session. Streaming APIs matter more for continuous-listening products; don't default to them just because they exist.
- **Multipart upload for 16kHz mono WAV**: OpenAI/Groq explicitly list `wav` as a supported format; Deepgram's docs reference `audio/wav` but didn't explicitly confirm 16kHz mono support in this pass — verify content-type/encoding requirements (Deepgram in particular sometimes wants explicit `encoding=linear16&sample_rate=16000` query params for raw PCM vs a self-describing WAV container) before wiring the adapter.
- **Pricing sources are largely third-party aggregator sites** (costgoat, diyai, tokenmix, pricepertoken, llmreference, toolradar) rather than primary provider pricing pages for several providers (Groq hourly rate, Deepgram batch rate, ElevenLabs, AssemblyAI, Speechmatics, Gladia) — cross-check against each provider's own `/pricing` page before finalizing cost models in the plan, since aggregator numbers can lag rate changes.

**Why:** Hark's plan is pivoting away from on-device STT to BYOK cloud STT; provider choice determines both the adapter architecture (how many distinct request/response shapes need code) and how the existing user-dictionary feature maps onto each provider's biasing mechanism, which was previously going to be sherpa-onnx hotwords (now abandoned per [[hark-stt-stack-risk]]).

**How to apply:** When writing the new plan/architecture doc, scope Phase 1 to two adapters (openai-compatible covering OpenAI+Groq, and deepgram) rather than building a generic N-provider abstraction upfront. Re-verify Groq/Deepgram real-world short-clip (2-15s) round-trip latency empirically during the spike — none of the marketing numbers found here (216x realtime, sub-300ms, 150ms) are apples-to-apples release-to-inject measurements for Hark's actual use case. Re-check Mistral Voxtral's and ElevenLabs' exact biasing param shape before writing their adapters (both flagged unconfirmed above).
